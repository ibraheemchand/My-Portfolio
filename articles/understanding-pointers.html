Understanding C++ Pointers: Memory Address to Dereferencing

Feb 15, 2024
8 min read
C++ Memory Management Pointers

🧠 Problem

While learning data structures, I kept getting segmentation faults when working with dynamic arrays. I didn't truly understand what pointers were doing under the hood. I could use them mechanically — declare with `*`, dereference with `*`, allocate with `new` — but couldn't visualize memory or predict when things would crash.

This lack of understanding made debugging incredibly frustrating. Every pointer error felt random.

⚠️ Error / Bug

Here's the error I kept hitting:

Segmentation fault (core dumped)

A simple example that would crash:

int* ptr = new int(42);
delete ptr;
std::cout << *ptr;  // CRASH! Accessing deleted memory

The program would segfault inconsistently. Sometimes it worked, sometimes it crashed. This randomness is what made it scary.

🔍 Root Cause

The issue was use-after-free: I was using pointers without understanding memory layout. After deleting memory, the pointer still held the address — but the memory was no longer valid.

Why did it randomly crash?

  • The freed memory might be reused by the OS for something else
  • Sometimes the old value was still there, so reads worked
  • Eventually, writes would hit a page that triggered protection checks

My mental model was wrong:

I thought pointers were "magical variables that point to things." I didn't grasp that:

  • A pointer is just a 64-bit integer (on 64-bit systems) storing a memory address
  • Stack memory is automatically freed when it goes out of scope
  • Heap memory persists until YOU explicitly call `delete`
  • Dereferencing an invalid pointer reads/writes to arbitrary memory

🛠️ Solution

I built a small project to visualize pointer behavior:

The Pointer Visualizer: A C++ program that prints memory addresses and values as I allocated and deallocated memory, then used GDB to step through and see exact memory state.

// 1. Stack vs Heap
int stackVar = 42;              // Lives on stack
int* heapVar = new int(42);     // Lives on heap

std::cout << &stackVar;       // Stack address ~0x7fff...
std::cout << heapVar;         // Heap address ~0x560...

// 2. Manual memory management
delete heapVar;
// Now heapVar still holds old address (dangling pointer)
// But dereferencing = undefined behavior

// 3. Solution: Smart pointers
std::unique_ptr<int> smartPtr(new int(42));
// Automatically deleted when smartPtr goes out of scope

Key realizations:

  • GDB is your friend: Use `print &variable` to see addresses, `print *pointer` to see values
  • Valgrind catches memory errors: Run `valgrind ./program` to detect use-after-free before crashes
  • Smart pointers (unique_ptr, shared_ptr) prevent these bugs: The memory is freed automatically

💡 Key Takeaway

Pointers are just variables that store memory addresses. Understanding stack vs heap is crucial. Always check if memory is still valid before dereferencing. Use tools (GDB, Valgrind) to catch errors early.

Best practices:

  • Prefer stack allocation when possible
  • If you must use heap, use smart pointers (unique_ptr, shared_ptr)
  • Never dereference after delete
  • Use RAII (Resource Acquisition Is Initialization) pattern
