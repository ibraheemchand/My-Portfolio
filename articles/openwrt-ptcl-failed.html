Why Installing OpenWrt on My PTCL ZTE H108N Failed - And What I Learned

📅 April 20, 2026 ⏱️ 13 min read ✍️ Hardware DIY
OpenWrt Networking Firmware Hardware DIY Troubleshooting
🧠 The Goal: A DIY NAS

I own an IP camera that streams 24/7. Every night, it needs to back up recorded footage somewhere safe, but I didn't want to:

  • Keep my laptop running all night (waste of electricity)
  • Buy a Raspberry Pi 3 (too expensive, overkill)
  • Pay for cloud storage (recurring costs)

My idea? Turn my PTCL ZTE ZXHN H108N modem into a NAS box. The modem is always on anyway, has a USB port, and theoretically supports file sharing. Perfect, right?

Spoiler: It wasn't.

The Root Problem: PTCL Locked Down Everything

My modem came with stock PTCL firmware that deliberately hides NAS features (FTP, SMB, DLNA) behind a locked interface. PTCL does this to maintain control and prevent customers from using unapproved hardware or configurations.

The features are physically built into the modem. They're just... hidden.

My Hardware: BCM63381B0 Broadcom Chipset

My modem has:

  • CPU: BCM63381B0 (Broadcom)
  • RAM: 64MB
  • Flash: 8MB
  • USB: USB 2.0 (one port)
  • Network: 4x Ethernet, WiFi 802.11b/g/n

This is a decade-old piece of hardware, but still plenty capable for NAS work.

⚠️ The Methods I Tried (And Failed)

I was offered 5-6 different approaches. Here's what happened with each.

Method 1: Telnet Unlock (The "Quick Fix")

What it promised: 30 minutes to unlock, non-destructive, temporary but functional

What I did:

git clone https://github.com/douniwan5788/zte_modem_tools
cd zte_modem_tools
pip install -r requirements.txt
python3 zte_factroymode.py --user admin --pass admin --ip 192.168.1.1 telnet open

I got telnet credentials. Success! I could SSH in. But then...

telnet 192.168.1.1
# Successfully logged in
upgradetest sdefconf 198
# Result: Command not recognized / No output / Did nothing

❌ Failed: The command either didn't exist, or PTCL's firmware variant doesn't respond to it. The modem didn't reboot, and the admin panel didn't unlock.

Why: PTCL uses a heavily modified firmware. The unlock command is known to work on standard ZTE firmware, but PTCL's custom build doesn't support it. The exact commands vary by firmware version.

Method 2: Alternative Telnet Commands

What I tried: Variations and alternative commands

sdefconf 198          # Didn't work
setmac upgradetest    # Didn't work
upgradetest           # Didn't work
setmac 4 198          # Didn't work

I even tried direct URLs in the browser:

http://192.168.1.1/admin/ftp.html
http://192.168.1.1/admin/samba.html
http://192.168.1.1/cgi-bin/webproc?getpage=/admin/ftp.html

❌ Failed: None of them worked. The URLs redirected to the main page or showed 404.

Lesson: There's no magic backdoor. If the ISP locks something, it's locked. Random commands won't fix it.

Method 3: Fresh Tomato Firmware Flash

What it promised: Full Broadcom support, user-friendly interface, built-in SMB/NAS

What I found: I discovered Tomato is great for Broadcom routers, but checks the device list:

  • Official Fresh Tomato device list: Not supported
  • Tomato by Shibby (older fork): Not supported
  • Advanced Tomato: Not supported

❌ Failed: Tomato only supports specific consumer router models. My PTCL ZTE modem wasn't on any list.

Why: Tomato developers need physical hardware to test. A obscure PTCL-customized ZTE modem isn't worth the effort. The device list is surprisingly short.

Method 4: DD-WRT (The "Most Compatible" Option)

What it promised: Widest router support, full NAS/SMB, large community

What I found: Searched DD-WRT's database at dd-wrt.com for:

  • "ZTE ZXHN H108N" → No match
  • "ZTE H108N" → No match
  • "BCM63381" → No match
  • "PTCL modem" → No match

❌ Failed: Even the widest-support firmware doesn't cover ISP-customized modems. Found forum posts from other PTCL users with the same issue.

Why: DD-WRT needs someone with your device to test. ISP variants are less popular, so fewer people test them.

Method 5: Generic OpenWrt brcm63xx Build

What it promised: Most flexible open-source firmware, broadest support, community-driven

What I considered: The oldest officially supported OpenWrt for brcm63xx: v15.05.1 (from 2016)

openwrt-15.05.1-brcm63xx-generic-96328avng-8M-flash-generic-squashfs-cfe.bin

❌ Didn't Flash (Would Have Failed): I was warned multiple times:

  • 10 years old → Security vulnerabilities
  • USB support → "Everything working except USB port" on similar devices
  • 8M flash → Tight fit for Samba + dependencies
  • No community proof → No one documented this device working

Why this would fail: Old firmware is obsolete and insecure. Effort to make it work isn't worth the risk.

5 different approaches. 0 successes. Each method failed for a different reason, but they all pointed to the same core problem: ISP firmware customization makes generic solutions incompatible.

🔍 The Root Cause: ISP Customization vs. Hardware Reality

PTCL's Firmware is Heavily Customized

Standard ZTE firmware would let me unlock features. But PTCL:

  • Modifies the boot process
  • Changes command structures
  • Removes factory mode access
  • Disables development features

This makes my modem incompatible with:

  • Generic unlock tools (they target standard ZTE, not PTCL variants)
  • Third-party firmware (they're not tested on PTCL variants)
  • Community fixes (too niche to document)

The Hardware vs. Software Mismatch

My modem's hardware is capable:

  • ✅ BCM63381B0 processor (Broadcom)
  • ✅ 64MB RAM (enough for SMB)
  • ✅ 8MB flash (enough with package removal)
  • ✅ USB 2.0 port (functional)

But the software is locked:

  • ❌ PTCL firmware disables telnet-based unlocking
  • ❌ Custom firmware variants aren't supported
  • ❌ Third-party firmware was never tested on this model

Why ISP Customization Exists

PTCL locks devices down because they need to:

  • Control what runs on your connection
  • Prevent customer support issues
  • Protect their network infrastructure
  • Maintain legal compliance

It's not malicious—it's business. But it's frustrating for tinkerers.

🛠️ What Would Actually Work

Option A: Find Pre-Flashed Stock Firmware (Non-PTCL)

If I could find original ZTE firmware without PTCL modifications, I could:

  1. Flash the stock firmware
  2. Use standard unlock methods
  3. Enable FTP/SMB features
  4. Have a working NAS

The problem: Finding this firmware is nearly impossible without ZTE support.

Option B: Accept Limitations & Use Stock Features

My PTCL firmware likely has SMB/FTP already built in. They're just disabled in the web UI. If I could unlock just the UI (or if PTCL had enabled it), I'd have my NAS instantly.

Option C: Hardware Modification (UART Serial)

I could:

  • Open the modem case
  • Connect via UART serial port
  • Access the bootloader
  • Unlock from there

The cost: Specialized tools, soldering skills, deep technical knowledge, and risk of permanent damage. Not worth a $0 NAS.

Option D: Buy Supported Hardware

If I had bought any of these instead:

  • ASUS RT-N66U (full Tomato/DD-WRT support)
  • Linksys WRT54GL (legendary support)
  • Any modern Netgear (partial support)

I could have flashed firmware in 15 minutes.

Option E: Continue Searching

Maybe the unlock command exists, and it's just not documented. Maybe someone else solved this. Maybe I'll find it in 6 months. But at that point, the modem might be replaced with newer hardware anyway.

💡 What I Actually Learned

1. ISP-Customized Hardware is Hard to Modify

When you get hardware from an ISP, they lock it down. It's intentional, comprehensive, and designed to resist modification.

2. Community Support is Everything

There are thousands of router models and dozens of firmware variants. The community only documents what people actually use. Rare combinations get abandoned.

3. Hardware Capability ≠ Software Support

Just because your modem has a Broadcom chip doesn't mean Tomato, DD-WRT, or OpenWrt will support it. The chipset is necessary but not sufficient.

4. Old Hardware Gets Abandoned

My ZTE H108N is from ~2013. It still works, but it's not worth developers' time. Newer hardware gets support; older hardware gets forgotten.

5. There's No Perfect $0 Solution for Every Device

Sometimes you need to spend money. Sometimes you need to accept limitations. Sometimes the $50 solution is better than the $0 failed attempt.

The Core Lesson: Hardware is not destiny. Your processor might be powerful, but your software might be locked. Your device might be technically capable, but its customized variant might not be supported. Sometimes the best solution is knowing when to stop and choosing a different path.

For Others Attempting This

If you're trying to modify an ISP-customized modem:

  • ✅ Do: Check if others with your ISP + model have succeeded
  • ✅ Do: Backup your current firmware first
  • ✅ Do: Know that you might fail (and that's okay)
  • ❌ Don't: Assume compatibility based on chipset alone
  • ❌ Don't: Keep trying random commands after several failures
  • ❌ Don't: Ignore warnings about unsupported devices

The Irony

My modem definitely has the capability to be a NAS. The hardware is there. The firmware likely has the features. But the ISP's lockdown is comprehensive enough to prevent my specific approach from working.

What's Next?

I'm not done with the DIY NAS idea. I have options:

  1. Find the magic PTCL unlock command (maybe it exists, just undocumented)
  2. Use my laptop's native NAS features (less ideal, but works)
  3. Eventually buy a cheap NAS device (when budget allows)
  4. Keep exploring OpenWrt for newer releases

For now, I've learned that sometimes the best solution is knowing when to stop.
